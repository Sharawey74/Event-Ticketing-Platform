13.1 TDD Workflow — Applied to Every Feature
Test-Driven Development (TDD) means writing the test before writing the implementation. This is not
just about coverage — it forces you to design the method's contract (inputs, outputs, exceptions) before
worrying about implementation details. The result is cleaner APIs and fewer edge-case bugs.
Why TDD for This Project Specifically
• The state machine has 11 states and 9 events — without tests written first, it is easy to wire a
transition incorrectly and not notice until a user's booking gets stuck. TDD makes every
transition a verifiable specification.
• The PricingEngine has 4 compounding rules — without a test for every combination, a
pricing bug would only surface in production when a user is overcharged or undercharged.
• The concurrency test is the only way to know the locking works — you cannot reason
about 100 simultaneous threads manually. The test is the specification.

TDD in Practice — calculateRefundAmount() Example
This walkthrough shows how TDD builds the calculateRefundAmount() method one behaviour at a
time:
1. Write the test: Write a test named
calculateRefundAmount_fiveDaysBeforeEvent_shouldReturn50Percent() annotated
@Test @DisplayName("calculateRefundAmount: should return 50% when cancelled 5
days before event"). Arrange: build a Booking with totalAmount =
BigDecimal.valueOf(100.00) and an Event whose startDate =
LocalDateTime.now().plusDays(5). Act: call
pricingEngine.calculateRefundAmount(booking, LocalDateTime.now()). Assert:
assertThat(refund).isEqualByComparingTo("50.00").
2. Run it — RED: Compilation error — calculateRefundAmount does not exist yet. Create a stub
that returns BigDecimal.ZERO. Now it compiles. Run again — assertion fails (0.00 ≠ 50.00). Still
RED — correct.
3. Implement — minimum: Add the 3-to-7-day branch: if daysUntilEvent >= 3 &&
daysUntilEvent < 7, return
booking.getTotalAmount().multiply(BigDecimal.valueOf(0.50)).
4. Run it — GREEN: Test passes. The 5-day case now returns 50.00.
5. Repeat for other branches: Write the ≥ 7-day test (full refund), then the < 3-day test (zero
refund). Each follows the same RED → GREEN cycle.

SECTION 15
Troubleshooting Guide
The 7 most common failure modes — each with symptom, root cause, diagnosis steps,
and fix
Every issue listed here was identified as a likely failure point during Phase 1A development. The
structure for each issue is: Symptom (what you observe), Root Cause (why it happens), Fix (exact
steps to resolve). Severity levels: HIGH = blocks all progress, MEDIUM = blocks a feature, LOW =
irritant but workable.
#1 — Stripe Webhook Not Receiving Events [ HIGH ]
Symptom Payment completes successfully on Stripe's hosted page but the booking remains in
PAYMENT_PENDING state. The confirmation page never loads.
Root Cause Stripe cannot reach localhost:8080 from the internet. Webhooks require a publicly
accessible URL or a local forwarding tool. Without forwarding, Stripe delivers the
event to a URL that points nowhere.
Fix 6. Option A — Stripe CLI (recommended): Install with brew install
stripe/stripe-cli/stripe. Login with stripe login. Run stripe
listen --forward-to localhost:8080/api/webhooks/stripe.
Copy the whsec_... signing secret printed to the console and set it
as STRIPE_WEBHOOK_SECRET in application-local.yml.
7. Option B — ngrok: Run ngrok http 8080. Copy the
https://xxxxx.ngrok.io URL. In Stripe Dashboard → Developers
→ Webhooks → Add endpoint, paste the URL with
/api/webhooks/stripe appended. Subscribe to events
checkout.session.completed and checkout.session.expired.
8. Verify: Stripe CLI prints each delivered event. After completing a test
payment, you should see checkout.session.completed appear in
the CLI output within 2 seconds.
Note The Stripe CLI is preferred because it handles the signing secret automatically and
does not require a Stripe Dashboard webhook endpoint configuration. Restart the
CLI whenever STRIPE_WEBHOOK_SECRET is not set correctly.
#2 — RabbitMQ Connection Refused on App Startup [ HIGH ]
Symptom Spring Boot fails to start with AmqpConnectException: Connection refused to
localhost:5672. All RabbitMQ-dependent features are broken.
Root Cause The RabbitMQ container is not running, is still starting, or did not pass its health
check before the app tried to connect.
Fix 9. Verify RabbitMQ container status: run docker ps | grep rabbit. If
the container is absent, run docker-compose up -d rabbitmq.

#3 — Redis Lock Deadlock — Lock Never Released [ HIGH ]
Symptom After a failed booking attempt, all subsequent reservations for the same tier
immediately fail with LockAcquisitionException. The tier appears permanently locked.
Root Cause The releaseLock() method was not called because the code crashed before reaching
the finally block, or the lock was acquired without a TTL (missing NX EX — lock
persists forever).
Fix 14. Inspect live lock keys: run redis-cli keys "seat:lock:*". If lock
keys are present, check their TTL with redis-cli ttl
"seat:lock:tier:1:user:123". A TTL of -1 means the key was set
without an expiry — this confirms the root cause.
15. Manually remove a stuck lock (dev only): redis-cli del
"seat:lock:tier:1:user:123".
16. Fix the root cause — verify executeWithLock() has a finally block: the
structure must be try { return operation.get(); } finally {
releaseLock(key, value); }. The finally block runs even if the operation
throws an exception.
17. Verify atomic acquisition: confirm the acquireLock() method uses
redisTemplate.opsForValue().setIfAbsent(key, value,
Duration.ofSeconds(ttl)) — the single atomic SET NX EX
command. If it uses separate SETNX and EXPIRE calls, replace them.
Note In redis-commander (http://localhost:8081), filter keys by seat:lock: prefix to monitor
locks in real time during development. All lock keys should disappear within 300
seconds even if never explicitly released, because the TTL acts as a dead-man's
switch.

#5 — @Transactional Not Rolling Back on External Service Failure [
MEDIUM ]
Symptom When a Stripe API call fails, the seat inventory in the database has already been
decremented and is not rolled back. Tickets are permanently lost.
Root Cause Spring @Transactional only rolls back changes to the Spring-managed DataSource.
Redis operations, Stripe API calls, and RabbitMQ publishes are external — they are
not part of the DB transaction and do not roll back automatically.
Fix 23. Understand the proxy bypass problem: @Transactional uses Spring
AOP proxies. If reserveTickets() calls reduceInventory() on the SAME
class, it goes through the same object (not the proxy) —
@Transactional on the inner method is silently ignored.
24. Fix — move the inner method to a separate @Service class:
BookingService injects InventoryService as a dependency. When
bookingService.reserveTickets() calls
inventoryService.releaseSeat(), it goes through the Spring
proxy — @Transactional applies.

#6 — Docker Compose Services Start Out of Order [ MEDIUM ]
Symptom Spring Boot starts before PostgreSQL, Redis, or RabbitMQ is ready. The app throws
Connection refused on startup and exits.
Root Cause Docker Compose depends_on without condition only waits for the container to
START, not for it to be ready. PostgreSQL takes ~3 seconds to accept connections
after starting. Spring Boot attempts connection within ~1 second of the container
starting.
Fix 27. Add condition: service_healthy to the app's depends_on in docker-
compose.yml:
28. depends_on: with sub-keys postgres: condition:
service_healthy, redis: condition: service_healthy,
rabbitmq: condition: service_healthy.
29. Verify each service has a healthcheck defined: PostgreSQL should
use pg_isready -U ticketing, Redis should use redis-cli ping,
RabbitMQ should use rabbitmq-diagnostics ping.
30. Add HikariCP retry to application.yml as a second safety net:
spring.datasource.hikari.connection-timeout: 30000 and
spring.datasource.hikari.initialization-fail-timeout:
60000. This gives HikariCP 60 seconds to acquire the first connection
before failing.
Note Run docker-compose up --build and observe the logs. The app container should print
"Waiting for postgres to be healthy..." before the Spring Boot startup banner appears.
If it starts immediately, the depends_on condition is not configured correctly.
