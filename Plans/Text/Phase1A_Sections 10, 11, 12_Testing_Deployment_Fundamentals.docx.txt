Testing Strategy
Test pyramid, unit test patterns, Testcontainers integration setup, coverage targets, and

Testing is not an afterthought in this project — it is woven into each day's work. The goal is confidence,
not coverage for its own sake. Every test must answer a real question: "Does this behaviour work?"
The pyramid below shows where effort is concentrated.

ℹ NOTE: Why 100% branch coverage on PricingEngine? It contains 4 independent pricing rules

General Unit Test Rules
• Use @DisplayName on every test with a human-readable sentence: "AVAILABLE → RESERVED:
should acquire Redis lock and decrement inventory". Test reports should be readable
without decoding method names.
• Use AssertJ (assertThat()) over JUnit assertEquals() — the fluent API produces far more
descriptive failure messages.
• For BigDecimal assertions, always use .isEqualByComparingTo("50.00") instead of
.isEqualTo() — the latter fails on scale differences (50.00 ≠ 50.0 with equals, but both pass
isEqualByComparingTo).

10.3 Testcontainers Integration Test Setup
All integration tests extend a single BaseIntegrationTest abstract class. This class starts one set of
Docker containers that are shared across all integration test methods in the test run — not restarted
per test. This is the critical optimisation that keeps the integration test suite from being unbearably slow.
BaseIntegrationTest.java — Class-Level Setup
• Class annotations: @SpringBootTest(webEnvironment =
SpringBootTest.WebEnvironment.RANDOM_PORT) — starts the full Spring context with a real
HTTP server on a random port; @Testcontainers — activates Testcontainers lifecycle
management; @ActiveProfiles("test") — loads application-test.yml which expects
Testcontainers-injected connection properties.
• Container fields are static — this is essential. Static containers are started once per JVM
process and shared by every test class that extends BaseIntegrationTest. Non-static
containers are started and stopped per test class, which is orders of magnitude slower.
Container Declarations
• PostgreSQL: static PostgreSQLContainer<?> postgres = new
PostgreSQLContainer<>("postgres:15-alpine") with
.withDatabaseName("ticketing_test"), .withUsername("test"), .withPassword("test").
• Redis: static GenericContainer<?> redis = new
GenericContainer<>("redis:7-alpine") with .withExposedPorts(6379) and
.waitingFor(Wait.forLogMessage(".*Ready to accept connections.*\n", 1)). The
waitingFor call ensures the container is fully started before Spring attempts connection —
without this, the app may connect before Redis is ready.
• RabbitMQ: static RabbitMQContainer rabbitmq = new
RabbitMQContainer("rabbitmq:3.12-management-alpine"). The Testcontainers
RabbitMQContainer provides helper methods getHost() and getMappedPort(5672).
@DynamicPropertySource — Wiring Containers to Spring
The @DynamicPropertySource static method injects the container's host and mapped port into Spring's
Environment before the application context is created. This overrides application-test.yml values at
test runtime. The registry adds these properties:

Using BaseIntegrationTest

every database change after each test — no manual cleanup needed.

JaCoCo Maven Plugin — pom.xml Configuration
Add the jacoco-maven-plugin version 0.8.11 to the <build><plugins> section of pom.xml. Three
executions are required:
• prepare-agent execution: Goal prepare-agent. No phase or configuration needed.
Instruments the bytecode so coverage data is collected during mvn test.
• report execution: Goal report, phase test. Generates the HTML report in
target/site/jacoco/ after tests run.
• check execution: Goal check. Configures the enforcement rule: element BUNDLE, counter
INSTRUCTION, value COVEREDRATIO, minimum 0.80. If coverage falls below 80%, ./mvnw
verify fails.

11.3 Dockerfile — Multi-Stage Build
The Dockerfile uses a two-stage build to produce a lean production image (JRE only, not JDK —
approximately 180MB vs 400MB). Dependencies are cached as a separate layer so rebuilds after
source changes do not re-download Maven artifacts.
Stage 1 — Builder (eclipse-temurin:17-jdk-alpine)
1. Copy Maven wrapper and pom.xml first (without src/): Copy .mvn/, mvnw, and pom.xml. Run
./mvnw dependency:go-offline -B. Because src/ has not been copied yet, Docker caches
this layer. On future rebuilds, if pom.xml has not changed, the dependency download is skipped
entirely.
2. Copy source and build: Copy src/. Run ./mvnw package -DskipTests -B. The
-DskipTests flag skips test execution in the Docker build (tests run separately in CI). The -B
flag enables non-interactive batch mode.
Stage 2 — Runtime (eclipse-temurin:17-jre-alpine)
3. Create non-root user: Run addgroup -S spring && adduser -S spring -G spring. Switch
to this user with USER spring:spring. Running as root inside a container is a security risk —
the non-root user limits blast radius if the container is compromised.
4. Copy the JAR: Use COPY --from=builder /app/target/*.jar app.jar. The
--from=builder flag pulls from the first stage. Only the JAR is copied — the JDK, Maven
cache, and source code are left behind.
5. Set JVM flags in ENTRYPOINT: java -XX:+UseContainerSupport
-XX:MaxRAMPercentage=75.0 -jar app.jar. -XX:+UseContainerSupport makes the JVM
respect the container's cgroup memory limit (not the host total). -XX:MaxRAMPercentage=75.0
allocates 75% of the container's memory to the JVM heap.

12.2 Design Ticketmaster (Week 2, Day 14 — 45 min timed)
Design a high-concurrency event ticketing platform handling peak loads (e.g. Taylor Swift tickets). This
is the system you are building in Phase 1A — you have direct insight into the trade-offs.
Architecture Components to Draw
• CDN: Serves static assets (event images, CSS, JS bundles). Reduces load on app servers for
90% of requests. Use CloudFront or Cloudflare.
• API Gateway: Rate limiting at the edge — critical for ticket rush. Limits each user to e.g. 10
requests/second to prevent bot flooding. Also handles auth token validation, request routing,
and HTTPS termination.
• Load Balancer → Auto-scaling App Server Pool: Routes requests across stateless app
server instances. Stateless is critical — any server can handle any request; session state lives
in Redis, not on the server.
• Booking Service (stateless): The core reservation API. Horizontally scalable. Acquires Redis
lock before touching inventory. The only state it holds is in the request context — nothing
persisted in-process.
• PostgreSQL Primary + 2 Read Replicas: Bookings and payments always go to the primary
(write path). Event browsing, category listing, and search queries go to read replicas (read
path). Replicas lag ~10ms behind primary — acceptable for read use cases.
• Redis Cluster (3 nodes): Distributed locks (seat reservation), cart data, tier availability counts,
event list cache. Three nodes provide availability — if one node fails, the cluster remains
operational.
• RabbitMQ: Decouples confirmation emails and PDF generation from the payment response. A
slow email provider cannot delay the checkout API response.
• S3 / Object Storage: Stores ticket PDFs generated after booking confirmation, event cover
images uploaded by organizers. Offloads binary storage from the database.

Bottleneck Analysis
• Primary bottleneck: The Redis lock acquisition is the critical path. Every reservation must
acquire a lock — this serialises access to each tier. At 50K concurrent users competing for the
same tier, 49,999 of them are blocked waiting. This is intentional — it prevents overselling —
but it means lock throughput is the ceiling.

• Secondary bottleneck: The PostgreSQL primary write path. 50K writes/second is near the limit

• What to mention in an interview: The seat reservation problem is fundamentally a distributed

availability (always accept requests), and latency (fast response). Redis gives us consistency +
low latency at the cost of availability if the cluster is down.

Trade-offs vs Ticketmaster Design
• Trade-off 1 — Consistency vs Availability for free RSVPs: Free events do not require a
distributed lock — the RSVP is cheaper. But if 2,000 people RSVP for a 100-person event, you
still need capacity enforcement. Use optimistic locking (@Version on the TicketTier
available_count) rather than Redis locks for free events — simpler, and the throughput
requirement is lower.
• Trade-off 2 — Organizer flexibility vs system complexity: Allowing organizers to configure
their own refund policies (per-event refund windows, percentage rules) requires a RefundPolicy
entity linked to each event, instead of the global 3-tier rule in Phase 1A. More flexible, more
complex — Phase 1B candidate.
• Trade-off 3 — Embeddable widget security vs feature richness: The embeddable widget
must be sandboxed (iframe with Content-Security-Policy) to prevent organizer websites from
intercepting payment data. This limits the widget to a redirect-to-Stripe flow — you cannot
embed Stripe Elements directly in the iframe without a more complex integration.
