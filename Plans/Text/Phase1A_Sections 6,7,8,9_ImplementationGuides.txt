6.2 Data Type Decisions — Why Each Type Was Chosen
Redis offers multiple data structures. Each was chosen for a specific capability required by the use
case:
Use Case Redis Type Why This Type
Distributed lock String SET NX EX is a single atomic command that sets a key only
if it does not exist and sets an expiry simultaneously. No
other data type offers this atomicity guarantee.
Inventory count String (int) Supports INCR and DECR — atomic integer operations that
prevent race conditions without external locking.
Cart data Hash Groups multiple fields (bookingId, tierId, quantity) under one
key. Allows reading individual fields without deserializing the
whole structure.
Event cache String (JSON) Full object serialized to JSON. Simple cache-aside — get,
set, delete. No field-level access needed.
Waitlist Sorted Set Scores enable ordered retrieval (FIFO by join timestamp).
ZPOPMIN atomically removes and returns the lowest-score
member — the earliest waiter.
Expiry marker String Only needs to exist (not hold a value). SET with EX gives a
self-cleaning key that the expiry job can detect by its
presence.
6.3 Distributed Lock — Acquisition and Release
acquireLock() — The Safe Atomic Pattern

The method: acquireLock(String lockKey, String lockValue, long ttlSeconds)
Critical Rule: Never use SETNX followed by a separate EXPIRE call. If the process crashes
between these two commands, the key is set with no expiry and the lock is held forever. The SET
key value NX EX ttl form is a single atomic command — it either sets the key with the TTL or does
nothing.
• Calls redisTemplate.opsForValue().setIfAbsent(lockKey, lockValue,
Duration.ofSeconds(ttlSeconds)), which maps to the atomic SET key value NX EX ttl
command.
• The lockValue is a UUID.randomUUID().toString() generated by the caller. This UUID is
the lock owner identity — it is the only value that can release this lock.
• Returns true if the lock was acquired (key did not exist and was set), false if the lock is
already held by another request. The caller should throw LockAcquisitionException on
false.
releaseLock() — Ownership-Checked Lua Script Release
The method: releaseLock(String lockKey, String lockValue)
⚠ WARNING: Never use a plain DEL command to release a lock. If your thread's TTL expired
and another thread acquired the same lock, DEL would delete their lock — not yours. The Lua
script prevents this.
The release uses a Lua script executed atomically by Redis. The script logic in plain English:
1. Read the current value of the lock key from Redis.
2. Compare it to the lockValue argument (the UUID this thread used when acquiring).
3. If they match: this thread still owns the lock → call DEL and return 1 (success).
4. If they do not match: another thread now owns the lock → return 0 (do not delete, no-op).
• Execute via redisTemplate.execute(new DefaultRedisScript<>(luaScript,
Long.class), Collections.singletonList(lockKey), lockValue).
• Returns true if the lock was successfully released (owned and deleted), false if the lock was
not owned by this thread (already expired or taken).
executeWithLock() — Safe Convenience Wrapper
The method: executeWithLock(String lockKey, long ttlSeconds, Supplier<T> operation)
• Generates a UUID as the lock value, then calls acquireLock(). Throws
LockAcquisitionException if the lock cannot be acquired.

6.4 Inventory Service — Redis-Backed Availability
The InventoryService keeps ticket availability counts in Redis for fast reads without hitting the
database on every request. The DB remains the source of truth — Redis is the fast cache layer.
Startup Warming
On application startup (@PostConstruct or ApplicationReadyEvent), the service reads
available_count for every TicketTier from the database and writes each value into Redis using
inventory:tier:{tierId}:available keys. This seeds the cache before any requests arrive.
Core Methods
• getAvailableCount(Long tierId) — Reads from inventory:tier:{tierId}:available in
Redis. On a cache miss (key absent), falls back to the DB and re-seeds the key. Returns the
current integer count.
• reserveSeat(Long tierId, Long userId, int quantity) — Called while the distributed
lock is held. Uses redisTemplate.opsForValue().decrement(key, quantity) for atomic
decrement. Returns false (and does not decrement) if the resulting count would go below zero.
• releaseSeat(Long tierId, int quantity) — Called by ReleaseSeatsAction on expiry or
cancellation. Uses redisTemplate.opsForValue().increment(key, quantity) for atomic
increment. After incrementing, also calls waitlistService.notifyNextInWaitlist().
• commitReservation(Long tierId, int quantity) — No-op in this architecture because the
DB available_count column is decremented directly when the Booking is persisted. If you
switch to a DB-as-write-behind strategy this method becomes the persistence point.

Spring Cache annotations (@Cacheable, @CacheEvict, @Caching) are used throughout EventService.
The RedisCacheManager is configured in RedisConfig.java with a default 10-minute TTL and per-

Cache Names and TTLs

EVENT_CACHE 10 min Evicted by key (event ID) on updateEvent() and

Eviction Rules — Service Layer Annotations
• getEventById(Long id) — annotated @Cacheable(value="EVENT_CACHE", key="#id"). On
hit: returns cached EventResponse without touching the DB. On miss: fetches from DB,
populates cache, returns result.
• updateEvent(Long eventId, ...) — annotated
@CacheEvict(value={"EVENT_CACHE","EVENT_LIST_CACHE"}, key="#eventId"). Evicts the
specific event entry and all list cache entries simultaneously.
• publishEvent(Long eventId) — annotated
@Caching(evict={@CacheEvict(value="EVENT_CACHE", key="#eventId"),
@CacheEvict(value="EVENT_LIST_CACHE", allEntries=true)}). Publishes an event
affects every filtered list, so allEntries=true wipes the entire list cache rather than guessing
which filter combinations are affected.
Why allEntries=true on lists? Event list results depend on many filter combinations (category,
city, date range, query string). It is impossible to know which cached filter combinations are affected
by a single event change. Wiping all list caches on any mutation is safe and simple — list data is
cheap to regenerate and has a short TTL anyway.
6.6 Waitlist — Redis Sorted Set Pattern
The waitlist uses a Redis Sorted Set at key waitlist:{eventId}:{tierId}. The score is the Unix
timestamp in milliseconds at the time the user joined. Lower score = joined earlier = served first (FIFO).
• joinWaitlist(Long eventId, Long tierId, Long userId) — Calls
redisTemplate.opsForZSet().add(key, userId.toString(),
System.currentTimeMillis()). If the userId is already in the set, the score is updated (re-join
refreshes position, which is acceptable).

6.7 RedisConfig.java — Bean Definitions
All Redis infrastructure is configured in common/config/RedisConfig.java. The following beans must
be declared:
• RedisTemplate<String, Object> — configured with StringRedisSerializer for keys and
Jackson2JsonRedisSerializer for values. This allows storing Java objects as JSON strings.
• RedisTemplate<String, String> — used specifically by DistributedLockService where
values are UUID strings that do not need JSON serialization.
• RedisCacheManager — built from RedisCacheConfiguration.defaultCacheConfig() with a
default TTL of 10 minutes, key serializer set to StringRedisSerializer, and value serializer
set to Jackson2JsonRedisSerializer. Per-cache TTL overrides are applied via
withInitialCacheConfigurations(Map<String,RedisCacheConfiguration>).
• Cache name constants class: define EVENT_CACHE, EVENT_LIST_CACHE, and
TIER_AVAILABILITY_CACHE as public static final String fields — used by @Cacheable and
@CacheEvict annotations to avoid hardcoded strings.
6.8 Debugging Redis in Development
Use redis-commander (included in docker-compose.yml at port 8081) to inspect all live keys. Key
things to check during development:
• Verify lock keys: Search for seat:lock:*. If a lock key exists with no TTL, the lock was
created without NX EX — this is a bug in the acquisition code.
• Verify availability counts: Search for inventory:tier:* and check that the values match
available_count in the DB after any test reservation.
• Verify cart expiry: Search for cart:* and check the TTL. It should be ≤ 300 seconds. If the
TTL is missing, the cart will never expire.
• Verify waitlist: Search for waitlist:* and use ZRANGE with WITHSCORES to view the
queued users and their join timestamps.

RabbitMQ Implementation Guide
Exchange topology, queue declarations, dead letter queues, message serialization, and
consumer error handling
RabbitMQ decouples post-payment processing from the critical path. Once a booking is confirmed, a
BookingConfirmedEvent is published to RabbitMQ and the API returns immediately. Email sending
and QR code generation happen asynchronously — a slow email provider cannot delay the
checkout response.

7.3 BookingEventPublisher — Sending Messages
The BookingEventPublisher component wraps RabbitTemplate with three named publish methods,
one per message type. Callers never interact with RabbitTemplate directly — they call the named
method with a typed POJO, and Jackson serializes it to JSON automatically.
• publishBookingConfirmation(BookingConfirmedEvent event) — calls
rabbitTemplate.convertAndSend(BOOKING_EXCHANGE, BOOKING_CONFIRMED_KEY, event).
Called from StripeWebhookController.handlePaymentSuccess() after the booking
transitions to CONFIRMED.
• publishEmailNotification(EmailNotificationEvent event) — calls
rabbitTemplate.convertAndSend(NOTIFICATION_EXCHANGE, EMAIL_SEND_KEY, event).
Used for general transactional emails (e.g. account registration, password reset).
• publishTicketGeneration(TicketGenerationEvent event) — calls
rabbitTemplate.convertAndSend(BOOKING_EXCHANGE, TICKET_GENERATE_KEY, event).
Triggers batch QR code generation for large group bookings where inline generation would be
too slow.
Message POJO Requirements
Each message class (BookingConfirmedEvent, EmailNotificationEvent,
TicketGenerationEvent, WaitlistAvailableEvent) must:
• Have a no-argument constructor (required by Jackson for deserialization on the consumer side).
• Be annotated with @Data and @Builder (Lombok) for construction and inspection.
• Contain all fields that the consumer needs — avoid lazy loading or DB calls inside listeners; the
message payload should be self-contained.
7.4 Consumer — Manual Acknowledgement and Error Handling
The BookingNotificationListener (and all other listeners) use manual acknowledgement rather
than Spring's default auto-ack. This ensures a message is only removed from the queue after it has
been fully processed.
Why Manual Acknowledgement?
• With auto-ack (the default), RabbitMQ removes the message from the queue as soon as it is
delivered to the consumer — even if the consumer crashes mid-processing. The message is
lost.

Handler Method Signature and Flow
The @RabbitListener method accepts three parameters:

• @Header(AmqpHeaders.DELIVERY_TAG) long deliveryTag — a unique identifier for this

Processing flow inside the handler:
5. Log entry: Log "Processing booking confirmation: bookingId={}" at INFO level using
@Slf4j.
6. Process: Fetch any additional details needed, generate PDF (optional for MVP), call
emailService.sendBookingConfirmation(event).
7. Acknowledge: On success, call channel.basicAck(deliveryTag, false). The second
argument false means "acknowledge only this message" (not all unacknowledged messages
up to this one).
8. Handle failure: In the catch(Exception e) block, log the error at ERROR level, then call
channel.basicNack(deliveryTag, false, false). The third argument false means "do
not requeue" — the message routes to the DLQ instead of being retried immediately (which
could cause infinite retry loops on persistent errors).
DLQ monitoring: Subscribe to the DLQ in Mailhog or the RabbitMQ Management UI (port 15672).
Any message that lands in the DLQ is a processing failure that needs investigation. In production,
set up an alert on DLQ depth.

Stripe Implementation Guide
Checkout session creation, webhook verification, idempotency, refund processing, and

The Stripe integration follows the Checkout + Webhook pattern: the backend creates a Checkout
Session and returns the hosted Stripe URL; payment happens on Stripe's page; a signed webhook
event notifies the backend of the outcome. The webhook is the only reliable signal — never trust a
redirect URL parameter to confirm payment.

8.3 StripeWebhookController — Webhook Handling
Located at POST /api/webhooks/stripe. The endpoint accepts a raw String body (not a
deserialized object) and the Stripe-Signature header. It must be excluded from CSRF protection in
SecurityConfig.
Step 1 — Signature Verification
• Call Webhook.constructEvent(payload, sigHeader, webhookSecret).
• If SignatureVerificationException is thrown: log at WARN level, return
ResponseEntity.badRequest().body("Invalid signature"). Never process an unsigned
event — this would allow anyone on the internet to fake payment confirmations.
• The @RequestBody String payload annotation must receive the raw bytes — do not use
@RequestBody with a deserialized type, as that would break the signature verification (Stripe
signs the exact byte sequence of the JSON payload).
⚠ WARNING: Do not read the request body twice. Spring's HttpServletRequest.getInputStream()
can only be read once. If any filter or interceptor reads the body before this controller, signature
verification will fail. Configure Spring Security to not consume the body for this endpoint.
Step 2 — Idempotency Check
• Check processedEventRepository.existsByStripeEventId(stripeEvent.getId()).
• If true: log at INFO and return ResponseEntity.ok("Already processed") immediately.
Stripe may deliver the same event more than once (network retries, delivery guarantees).
Processing twice would double-confirm a booking, generate duplicate QR codes, and send
duplicate emails.
• The ProcessedStripeEvent table needs a unique index on stripe_event_id at the DB level
as a second guard.

SECTION 9
State Machine Implementation Guide
BookingStateMachineConfig, stateless-per-request wrapper, all transitions, guards,
actions, and test patterns
The Spring State Machine defines and enforces the legal state transitions for every Booking. Without it,
a developer could accidentally confirm a booking that was never reserved, or approve a refund on an
expired booking. The state machine makes illegal transitions compile-time safe and runtime-
enforced.
9.1 State and Event Enums
BookingState — All 11 States
State Meaning
AVAILABLE Initial state. Tickets exist but no booking has been initiated. This is a virtual state —
no Booking entity exists in this state; it represents the absence of a booking.
RESERVED User added tickets to cart. Seats are locked in Redis. Booking entity created with
expires_at = now + 5 minutes. The 5-minute countdown timer starts.
PAYMENT_PENDING User proceeded to checkout. Stripe Checkout Session created. Booking is awaiting
the webhook confirmation from Stripe.
CONFIRMED Payment received (webhook confirmed). QR codes generated for all tickets.
Booking confirmation email sent via RabbitMQ.
ATTENDED User checked in at the event door. QR code was scanned and verified. Terminal
happy-path state.
EXPIRED Reservation timer ran out (5 minutes elapsed without checkout). Seats
automatically released back to inventory. User must start over.
RELEASED Seats released back to inventory after a PAYMENT_FAILED or EXPIRED
transition. Internal cleanup state — not directly shown in the UI.
PAYMENT_FAILED Stripe reported payment failure or session expiry. Seats released. User sees a
payment failure message.
REFUND_REQUESTED User cancelled a CONFIRMED booking. Refund eligibility checked by
PricingEngine. Awaiting processing.
REFUND_APPROVED Stripe refund processed successfully. Seats released, waitlist notified. Terminal
state for cancelled bookings.
REFUND_DENIED Cancellation requested but refund policy gives 0% refund (< 3 days before event).
Booking remains in this state — no seats released.

9.3 Guards and Actions
Guards — Conditions That Block Transitions
Guards implement Guard<BookingState, BookingEvent> and return true to allow the transition or
false to block it. If a guard returns false, the state machine rejects the event without throwing an
exception by default — your wrapper must check the return value of sm.sendEvent() and throw
InvalidStateTransitionException explicitly if it returns false.
• SeatsAvailableGuard: Implements Guard<BookingState, BookingEvent>. Reads tierId
from the message header, calls inventoryService.getAvailableCount(tierId), and
returns count > 0. This is the gate that prevents overselling before the Redis lock is even
acquired.
• BookingOwnedByUserGuard: Reads bookingId and userId from message headers. Calls
bookingRepository.existsByIdAndUserId(bookingId, userId). Returns true only if the
requesting user owns the booking. Used on REQUEST_REFUND and cancellation transitions.
Actions — Side Effects on Transitions
Actions implement Action<BookingState, BookingEvent>. They fire synchronously during the
transition. All actions have access to the StateContext which carries message headers and the
extended state variables — this is how they retrieve bookingId and other context values.
Action Class What It Does
StartReservationTimerAction Sets booking.expires_at = LocalDateTime.now().plusMinutes(5) and
saves the booking. Also writes the reservation:expiry:{bookingId}
key to Redis with a 300-second TTL. The ReservationExpirationJob
detects this key when it runs every 30 seconds.

9.4 BookingStateMachineConfig.java — Structure
The config class is annotated @Configuration, @EnableStateMachineFactory, and @Slf4j. It
extends StateMachineConfigurerAdapter<BookingState, BookingEvent> and overrides three
methods:
• configure(StateMachineStateConfigurer): Declares the initial state
(BookingState.AVAILABLE), the end state (BookingState.ARCHIVED), and all states via
EnumSet.allOf(BookingState.class).
• configure(StateMachineTransitionConfigurer): Chains all 11 transitions using the fluent
builder pattern. Each transition uses .withExternal() (state changes between different states)
with .source(), .target(), .event(), optional .guard(), and optional .action() calls.
Transitions are chained with .and().
• configure(StateMachineConfigurationConfigurer): Sets .autoStartup(false) (the
machine is started manually per request in the service wrapper, not globally on startup).
Registers a StateMachineListenerAdapter that logs every state transition at INFO level using
SLF4J.
@EnableStateMachineFactory vs @EnableStateMachine: Use @EnableStateMachineFactory
(not @EnableStateMachine) because the stateless-per-request pattern requires a new
StateMachine instance per request, retrieved from the factory. @EnableStateMachine creates a
single shared instance which would have race conditions under concurrent load.

Why Stateless-Per-Request?
• Spring State Machine is not inherently thread-safe when shared. If a single machine instance is
reused across concurrent requests, two threads could simultaneously attempt transitions and
corrupt each other's state.
• The stateless-per-request pattern solves this: for every sendEvent() call, a fresh StateMachine
instance is obtained from the factory, its state is restored from the database, the event is
processed, and the resulting state is persisted back to the database. The machine is then
discarded.
• The DB is the authoritative state store. Redis, the machine instance, and all in-memory objects
are ephemeral.
sendEvent() — Step-by-Step
36. Get machine from factory: Call
stateMachineFactory.getStateMachine(bookingId.toString()). The booking ID is used
as the machine ID for logging purposes.
37. Restore current state: Fetch the Booking entity from the DB. Call
sm.getStateMachineAccessor().doWithRegion(access ->
access.resetStateMachine(new DefaultStateMachineContext<>(booking.getState(),
null, null, null))). This forces the machine to the state currently stored in the DB,
discarding any prior in-memory state.
38. Start the machine: Call sm.start().
39. Inject context variables: Call
sm.getExtendedState().getVariables().put("bookingId", bookingId) and any other
context values (tierId, userId, quantity) that actions and guards need during the transition.
40. Build and send the message: Build
MessageBuilder.withPayload(event).setHeader("bookingId", bookingId).build().
Call boolean accepted = sm.sendEvent(message)
41. Check acceptance: If !accepted: throw InvalidStateTransitionException("Cannot send
event " + event + " in state " + booking.getState()).
42. Persist new state: Read sm.getState().getId() and call booking.setState(newState) +
bookingRepository.save(booking).
43. Stop the machine: Call sm.stop(). Releases resources.
✔ TIP: Wrap sendEvent() calls in @Transactional at the service layer (e.g. BookingService,
PaymentService). This ensures that if the DB save of the new state fails, the whole operation rolls
back — you never end up with a state machine that transitioned but a DB that still shows the old
state.

9.7 Common State Machine Mistakes
Mistake Correct Approach
Using @EnableStateMachine Use @EnableStateMachineFactory always. The per-request pattern
(singleton) instead of requires a fresh instance per sendEvent() call — a singleton would have
@EnableStateMachineFactory concurrent state corruption under load.
Not checking sm.sendEvent() Always check the boolean return. If sendEvent() returns false, the transition
return value was rejected (guard returned false or illegal transition). Throw
InvalidStateTransitionException explicitly.
Not calling sm.stop() after use Call sm.stop() in a finally block. Failure to stop leaks resources and can
cause the machine to appear "still running" in logs.
Calling sendEvent() without Always call doWithRegion + resetStateMachine before sm.start(). Without
first restoring state via this, the machine starts in its initial state (AVAILABLE) instead of the
resetStateMachine() booking's current DB state.
Putting business logic inside Keep BookingStateMachineConfig purely declarative. All logic belongs in
the Config class the Guard and Action beans — which are separate @Component classes
injected into the config.
Not using @Transactional If the DB save of the new state fails, the action (e.g. QR generation) has
around sendEvent() calls already run but the state is not persisted. Wrap the entire sendEvent() +
bookingRepository.save() sequence in a @Transactional boundary.
